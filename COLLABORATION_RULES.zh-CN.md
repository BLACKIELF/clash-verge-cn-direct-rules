# 通用协作习惯

## 0701v1

较上一版：新增通用协作规则文档，覆盖第一性原理、对抗式审查、分层修改、写入/发布确认、版本说明、证据优先、最小验证、Markdown 输出、模板沉淀和 Clash / 网络规则适配。

## 核心规则

每次开始任务前，先判断真正问题、必须结果、可选偏好和最小有效动作。

执行前做对抗式审查：需求是否误解、文件是否会改错、是否覆盖已有成果、是否把临时规则写成全局规则、是否缺验证、是否只修表面症状。

先判断修改层级，只改用户指出的层：规则、文件、配置、输出格式、安装/更新、网络/环境、版本管理、示例模板。

默认先在对话里记录。只有用户明确要求写入文件、写进 skill、保存到规则、同步本地、刷新插件或更新包，才写文件。

推送 GitHub、刷新插件、打包分享包前必须确认。若同一条用户消息已明确要求执行，则按该请求执行，并在推送前给出变更范围。

正式更新使用版本号 `MMDDvN`，例如 `0701v1`。每次写较上一版的更新说明，最新放最上面。

没有实际读取文件，不说已查文件。没有实际运行验证，不说已验证成功。

配置、安装、网络、脚本、规则类任务，修改后必须做最小验证。YAML 检查语法；Clash 规则检查端口、DNS、连接或路由；GitHub 更新看 status/log；Skill 更新检查触发名和结构。

Markdown 默认紧凑、可复制、少废话：不连续空两行，代码块内不连续空行，可复制内容放独立代码框，避免过度嵌套列表。

已验证成功的模板不要轻易推翻。新规则进入长期 skill 前，先分成通用规则、项目专用规则、临时执行写法、成功样本、失败修复经验。

优先用正向规则锁定正确结果。只有出现具体错误后，再补对应负面限制。

处理 Clash、代理、联网规则时，先判断是 DNS、代理端口、规则匹配、进程、环境变量还是目标网站本身。不要只看配置文件就说成功。修改前备份或说明改哪段，修改后验证 YAML，再验证端口、连接、路由或 curl 结果。

优先做最小可验证改动。能改一行，不改十行；能补一条规则，不重写整份配置；能先对比差异，不直接覆盖。
